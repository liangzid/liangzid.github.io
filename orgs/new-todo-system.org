最近又重新思考了一下如何进行个人工作的版本流控制，对之前的reflection思路进行了一波改造升级。

改进的原则是：

+ 实用大于美观。不过度追求所覆盖的全面和完整，只聚焦在最重要的几个方向。其余的则放空
+ 强调可持续性。
  + 提供退出机制，以应对较为简单的个人记录。
  + 提供模板，以方便插入
  + 在文件管理与版本控制上也突出可持续。不会出现过往todo list过于复杂的情况
+ 与当前工作流深度嵌入
  + Git集成与全端浏览
  + Emacs + Org Mode集成
+ 诚实与robustness
  + 仓库私有
  + 扩展性强

目前新版的个人版本控制流程如下。

可以看出，整个项目仅仅包含四个方面，分别是周报（weekly），经济、身体、外貌。至于其他，比如社交等等暂时没有加进去；阅读之类的也没有显式地加进去。这不是因为后面二者并不重要，而是他们并不需要非常严格的长期监控与处理。

对于所关注的四个方面，只有weekly是需要日常维护的，剩余的如appearance、body、finance都可以以一个较低的频率进行更新。reflection则需要每周更新。具体来说是：

+ Weekly：将一周七天的所有计划纳入一个文件中进行管理。
+ Appearance：负责管理我的外在，维持作为一个正常人类的形象
  + Cloth：对我的衣服进行管理。定期更新、删除、修改。
  + Haircut：对我的头发进行管理。定期修理。
+ Body：负责监控我的身体状况。与Appearance一样，我认为这也是仍然需要重点关注的点。
  + Physical：指个人的身体素质。包括：体重、体脂等个人数据。从每周的周报中统计进行各项体育运动、健身等等的强度，以构成反馈。
  + Sexual-Love：用于监控个人的性压抑程度。记录自己的性行为次数、频率、相关人员、时间。由此进行后续的心理健康状况分析。
+ Finance：记录自己的经济状况。
  + Monthly：每个月都记录自己的支出和收益，挖掘出不合理的支出。设定消费数目，对金钱进行合理的分配。
  + Yearly：从全年的角度制定每一年的经济计划 （如学生时期可能以尽量产生赤字为主，多体验）。并设置每个方向的消费比重。
+ Reflection
  + Weekly：基于每周周日志的文件对本周状态进行总结。 聚焦在细节。
  + Monthly：对于每个月的情况进行总结。 每个月都应该让事态产生明显变化。
  + Quarterly：对于每个季度进行总结。 每个季度都应该完成一个战略目标点。
  + Yearly：对全年的总结。 这应该站在一个比较高的层面进行总结。
+ Inbox：用于记录某些灵光。需要被定期清理。


* 一个理想的运行结构

1. 每日开始，应该打开当周的weekly，撰写上一日的reflection
2. 应排列今日的TODO，并按照流程逐步进行。
3. 每周周五晚上或周六上午，应该进行每周的reflection，并整理手机上的身体数据。
4. 每个月发工资时，对于这个月的银行流水进行总结和记录。检查银行卡、微信、支付宝等。
5. 每个月月末阅读该月的relfection，然后撰写月度总结。
6. 每次和导师开完会对appearance的情况进行一次更新。
7. 每个季度撰写反思和总结。
8. 每年都对全年的情况进行一次系统的分析。可以把语料发给AI。


维持持续记录的要点：
1. 在周一、二 等日安排攻坚的难题。
2. research等tag的TODO ITEM个数应该与body、fiction、relax等实现平衡。
3. 允许在某些特殊天里“潦草记录”。但是认为连续性最为重要。
4. 允许补录。允许潦草补录。
